# Architecture

RepoScryer is organized as a Rust workspace with strict responsibility boundaries.

- `reposcryer-core`: shared data model, stable identifiers, index run types, and incremental plan types
- `reposcryer-config`: `.reposcryer/config.toml` loading, validation, defaults, and default file generation
- `reposcryer-ingest`: file scanning, filtering, hashing, skip reporting, and language detection
- `reposcryer-parser`: heuristic language adapters, parser registry, warning conversion, raw imports, and symbol chunks
- `reposcryer-graph`: Phase 1 export graph assembly from scanned files and parsed output
- `reposcryer-store`: Kuzu connection management, schema initialization, graph persistence, index run tracking, file fingerprints, incremental status queries, and derived file dependency edges
- `reposcryer-export`: artifact generation for graph, symbols, warnings, and repo map
- `reposcryer-cli`: command orchestration and human-readable summaries

## Phase 3 Index Flow

1. Load `.reposcryer/config.toml`, or use defaults if it does not exist.
2. Scan the repo root into a `ScanResult` using the loaded config.
3. Open the configured Kuzu output path through `KuzuGraphStore`.
4. Initialize schema and create an `IndexRun`.
5. Compare scanned fingerprints with Kuzu `File` nodes for the current `IndexScope`.
6. Parse only `Added`, `Modified`, and `ReindexNeeded` files.
7. Replace each changed file subgraph idempotently.
8. Soft-delete `Deleted` files.
9. Rebuild derived `File IMPORTS_FILE File` edges for explicit local imports.
10. Complete or fail the `IndexRun`.
11. Write `state.json` and Phase 1 export artifacts under the configured output directory.

CLI code does not issue Kuzu queries directly. It orchestrates `reposcryer-ingest`, `reposcryer-parser`, `reposcryer-store`, and `reposcryer-export`.

## Configuration

Config discovery is fixed to `<repo>/.reposcryer/config.toml`. The config crate validates TOML, merges user ignored directories with default safety ignores, and exposes a typed `RepoScryerConfig` to ingest, export, and CLI orchestration. See [configuration.md](configuration.md) for the schema.

## Store Model

Kuzu is the single source of truth for graph data and incremental state. The graph schema includes `Project`, `Worktree`, `IndexScope`, `File`, `Symbol`, `Import`, `Chunk`, `Warning`, and `IndexRun` nodes, plus relationships for containment, definitions, chunks, warnings, touched files, and resolved file imports.

Phase 3 keeps project/worktree/scope identity simple and local to the current repo path. `file_id` is scope-bound using `project_id + scope_id + relative_path`. Branch-aware and workspace-aware scope selection is deferred.

## File Dependencies

`reposcryer-store` derives `IMPORTS_FILE` edges from stored `Import` nodes and active `File` nodes. The current resolver is intentionally narrow: it resolves Rust module paths such as `mod auth;`, `pub mod services;`, `use auth::AuthService;`, `use crate::services::auth::AuthService;`, `use self::token::Token;`, and `use super::db::Database;` to indexed local module files when the path evidence is explicit.

Rust resolution prefers the deepest matching module prefix. For example, `crate::services::auth::AuthService` resolves to `src/services/auth.rs` before falling back to `src/services/mod.rs`. `self::` starts from the current module directory, `super::` walks to ancestor module directories, and `crate::` starts from `src`. Standard library roots such as `std`, `core`, and `alloc` are ignored.

Graph query APIs are also owned by `reposcryer-store`:

- `file_neighbors` returns direct outgoing dependencies and direct incoming reverse dependencies.
- `file_impact` performs transitive reverse traversal over `IMPORTS_FILE` edges to identify files that may be affected by a file change.
- `scope_graph_summary` reports current Kuzu scope counts for active files, deleted files, symbols, imports, dependency edges, warnings, index runs, and the latest run status.

This is not a call graph. No `CALLS` edges are generated by the Phase 3 implementation.
